# ADR-0001: Alpha Asset Model

> **Status:** Accepted · 2026-03-02 · Owners: FPC maintainers
> **Normative source:** [docs/spec/adr-0001-alpha-asset-model.md](https://github.com/NethermindEth/aztec-fpc/blob/main/docs/spec/adr-0001-alpha-asset-model.md)

## Context

Prior to this ADR, repository and product docs had mixed two incompatible models:

1. **Single multi-asset contract deployment** — one FPC, many tokens via on-contract allowlist
2. **One FPC deployment per accepted asset** — a stack per token

This ambiguity produced conflicting acceptance criteria, operator guidance, and implementation plans.

## Decision

**For Alpha, the product model is single-deployment multi-asset.**

- One FPC deployment handles multiple accepted assets.
- "USDC + ETH variants" is satisfied only when a single deployed contract instance can accept at least one USDC variant and at least one ETH variant **without redeploying** the contract for each asset.
- Deploying separate single-asset stacks per token is an interim workaround and does **not** satisfy Alpha done-criteria.

## What this changes

### Contract (on-chain)
- Multi-asset storage/model (allowlist or equivalent) in each contract.
- Admin controls for asset lifecycle (add/remove), with clear governance constraints.
- Quote binding remains user-specific **and** asset-specific — the selected payment asset is in the quote preimage.

See [FPCMultiAsset](/contracts/fpc-multi-asset) for how this manifests in the deployed contract.

### Attestation service
- One attestation instance per contract deployment supports multiple assets and asset-specific rates.
- Quote API includes the selected payment asset in request and response.
- Operator config supports per-asset pricing and policy — see [`supported_assets`](/operations/configuration#assets).

### Top-up service
- Top-up remains keyed by fee-payer contract address (not per-asset).
- No per-asset top-up split for Fee Juice, but sizing must account for aggregate traffic across all accepted assets.

### Wallet integration
- Wallet discovery targets one contract endpoint plus its supported asset set — see [Wallet Discovery](/reference/wallet-discovery).
- Wallet quote flow selects an asset and requests a quote for that asset.

## Consequences

| Consequence | Implication |
|---|---|
| ✅ Aligns Alpha requirements with desired UX | One operator endpoint supporting common bridged assets |
| ⚠ Increases implementation scope | Multi-asset support is more than per-deployment single-asset |
| ⚠ Requires migration path | Operators running interim multi-instance setups need to move to the single-deployment model |
